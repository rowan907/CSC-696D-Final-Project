\subsection{Problem Description}
In this project, we created several novel visual analytic systems for git
data. The visualizations we propose help developers quickly gain an understanding of
git repositories. In particular, we seek to give users an understanding of how a
repository changes over time. We focus on a few key questions.
\begin{enumerate}
    \item Who are the main contributors to a repository?
    \item What were the development priorities for a repository at any given
        time?
    \item Can we identify a time when AI-assisted development became popular on
        a given repository?
    \item What is the best visualization for giving developers an understanding of the
        evolution of project structure over time?
\end{enumerate}
\subsection{Motivation}
Developers often work with large amounts of code and are forced to quickly learn
the customs and conventions of many repositories; this process can be very
time-consuming. By giving developers tools for quickly visualizing the history
of a repository, we can immediately give them an understanding of conventions, key
contributors, and etiquette. \par
Many visualizations have been developed over the years to work with git data;
in this project, we developed a handful of new visualizations by first
studying visualizations that others have developed, then either refining them to
improve their visual clarity or augmenting them with additional data. \par
We developed 3 visualizations for this project.
\begin{enumerate}
    \item A commit graph showing branches of the repository. Users may filter by
        selecting portions of commits for the graphs below to use. In a workflow, this
        focuses on examining a portion of commits to see differences in lines of code, authorship,
        files, and commit messages. Users may also filter by authors. This project
        emphasizes authorship as described in Git-Truck's user-evaluated metrics.
    \item A `stabilized' Sediment Graph based on the work of
        Erik Bernhardsson \cite{gitoftheseus}.
        Bernhardsson proposes a unique way to understand the age of the contents of a
        repository. We improved this design by changing the stacking behavior so that as rows
        are added, the graph grows in both the positive and negative $y$ directions.
        This improves the stability of the graph by keeping
        individual lines of sediment more horizontal. Reducing the visual
        interference caused by the sediment layers traveling diagonally up the
        graph makes their width easier to determine.
        We also, based on feedback we received from our peers, added two new views to this
        visualization.
        In addition to dividing the layers of sediment by code
        age, we also provided the option to divide the sediment by author,
        supporting both total lines of code by author and number of commits by
        author. These views may be an even better use case than code age, as
        authorship is generally a more important metric.

        In addition, we added interactions to this graph. First, hovering on each sediment
        gives users information on the time range, authorship, or number of commits. Rather
        than having users estimate the values for each of these, we provide them on hover for
        visual clarity. We also allow users to extract sediments from the graph, particularly to
        remove outliers. For example, if one author has a large number of lines of code written,
        that author can be extracted to normalize the distribution of all other authors. Users may
        use this to increase the expressivity of visual area and also to find
        the top contributors of the repository.
\item Word Cloud-style view that allows users to select
        many commits via a time window selection and see a word cloud of
        the commit messages. This allows users to quickly see the
        development focuses of a repository at a particular time window. This is
        inspired by a view from \cite{githru}. We seek to enhance their view by adding a time
        window selection allowing these clouds to be generated for any section
        of the commit history. We also provide users the ability to click a word
        from the word cloud and highlight the commits that include that
        word, allowing users to quickly see what changes dealt with certain
        topics.

        We augment the Word Cloud through simple tf-idf weighting of terms.
        Rather than having users sift through all terms manually, we attempt to
        provide important, unique terms to sift through. At times, authors and
        referenced libraries are identified, providing a higher level of understanding
        than the most frequently used terms alone would provide.
    \item File Cluster Map. A grouping-based visualization based on a force-directed
        graph that clusters files or directories that
        are often modified together, this helps users understand which parts
        of the codebase are most tightly coupled. We animated this view to show
        the evolution of the repo over time and coupled it with a timeline view
        allowing users to scrub to particular sections of the animation. 

        Nodes here correspond to both files and directories. Directories are opaque nodes,
        and files are transparent nodes with dotted outlines. Directories can be expanded 
        to reveal their files, and files can be collapsed back into directories. The directory view gives users
        the ability to focus on higher-level abstract details. For instance, is documentation
        being created alongside tests? With time animation, this becomes apparent for users.

        With all node-link diagrams, the number of links dramatically impacts the visual
        expressivity of the graph. This project's File Cluster Map is no different. Therefore,
        we allow users to change the number of files shown when a folder is expanded. The
        files are selected by the number of lines of code they contain in order to select
        the most important documents.
\end{enumerate}
\subsection{Related Research Papers}
\begin{itemize}
\item \cite{githru} ---
    This work shows an all-in-one visual analytic tool for examining git history
        and provides a lot of great background on git. Many techniques here may
        be useful; in particular, the paper provides a lot of detail on the
        technical aspects of creating visualizations with git metadata. We
        also took inspiration from this paper for our second proposed
        visualization as discussed above.
\item \cite{gittruck} --- Git-Truck is a hierarchical file-organization-oriented
    visualization tool that represents git repositories as different treemaps.
    Each treemap has its nodes sized by the number of lines of code in the repository,
    and the color of its nodes can represent a variety of interesting factors
    such as number of commits, top contributors, or the last change date.
    This visualization shows the current structure of a repository, rather than the temporal
    focus we have in this project. Mainly, we emphasize the importance of seeing
    the historical development of features in a repository. Git-Truck also provides a
    set of metrics that provide guidance for this paper's implementation: number of commits,
    single author, and top contributor. This project highlights the importance of authorship for
    repository exploration.
\item \cite{changeprism} --- This paper gives another approach to visualizing
        code changes that relies on semantic analysis. Though this paper's focus
        is more on visualizing individual commits and discrete changes rather
        than the type of broad analysis of larger repositories we did,
        it provides some good background on existing visual analytic techniques.
\end{itemize}
\subsection{Technical Approach}
Our technical approach is, at a high level:
\begin{enumerate}
    \item Collect \texttt{.git} files containing git metadata from repositories we wish to
        study.
    \item Use a pre-processing script to execute git commands to query this data
        and create a \texttt{.json} file for each repository containing all commits we
        want to analyze and any useful metadata like branch, parent, files
        changed, etc.
    \item Load our \texttt{.json} files at run time when the user selects a
        particular repository they want to analyze.
    \item Build our visualizations from the processed data.
\end{enumerate}
Our pre-processing script lives inside the project
(\texttt{app/scripts/preprocessgit.mjs}) and can be triggered manually
with \texttt{npm run preprocess:git}. The script uses JavaScript's \texttt{exec}
function to run git with the flag \texttt{--git-dir=[.git file]}; this command
allows git to behave for most commands as if it's inside the project we're
targeting despite only having access to the metadata. This saves us from having to
pull the whole project we wish to study. \par
We then run a \texttt{git log} to get all commit hashes across all branches (for
performance reasons we cap this at the 4500 most recent commits when more are
available). Finally, we run git commands on every individual commit to get
information like predecessor and branch. The information we pull from each
commit is below and is used to support all three visualizations. \par

\subsubsection*{File Cluster Map}
The File Cluster Map is built on a D3 force-directed graph.
A graph-construction pass iterates over all commits: for each commit touching
50 or fewer files, every pair of co-modified files receives a weighted edge
whose weight counts the number of times that pair appears together across all
commits; edges with weight below 2 are discarded.
Files beyond a configurable per-directory cap are collapsed into their parent
directory node, keeping the graph readable by default while still allowing
users to expand directories on demand.
Node radius is scaled by total lines of code using a square-root scale
(capped between 5 and 28 px), so the most-touched files are visually
prominent without dominating the layout.

The force simulation uses four forces: a charge repulsion of $-180$ to spread
nodes apart, a centering force, a link force whose strength is proportional
to co-occurrence weight, and a collision force using each node's radius.
Directory nodes are rendered as solid filled circles; file nodes use a dashed
stroke outline to visually distinguish the two levels of the hierarchy.
Link width is also scaled by co-occurrence weight, reinforcing the coupling
signal through both color and thickness.

To animate the graph over time, the component is driven by a filtered commit
array from the timeline brush: as the brush window advances, the graph is
rebuilt from only the commits in that window, and the force simulation is
restarted.
A stable color map assigns each top-level directory a consistent hue across
all animation frames so that clusters remain visually identifiable as the
graph evolves.

\subsubsection*{Sediment (Stream) Graph}
The Sediment Graph is implemented using D3's stack generator with three
selectable data modes.
In \textit{era mode}, the commit timeline is divided into 32 equal cohorts;
each file is attributed to the cohort in which it was first introduced, and
deletions erode the surviving line count of that birth cohort.
In \textit{author lines-of-code mode}, each file is owned by the first among
the top-50 authors (by total additions) to touch it, with deletions eroding
that author's count; in \textit{author commits mode}, cumulative commit
counts per author are plotted instead of lines of code.
All three modes produce a matrix of 200 evenly spaced time samples by
cohort, which is fed to the D3 stack generator.

The key design choice distinguishing our graph from Bernhardsson's original
is the stacking offset.
We use \texttt{stackOffsetWiggle} with \texttt{stackOrderInsideOut}, which
centers the stack around a horizontal baseline and orders bands to minimize
visual slope.
This keeps individual sediment bands close to horizontal regardless of their
relative magnitude, making width (and therefore surviving line count) easier
to read than in a bottom-anchored stack where upper bands travel diagonally
across the chart.

Paths are rendered with \texttt{curveBasis} smoothing and colored by HSL
values computed from cohort index.
Hovering a band dims all others to 0.3 opacity and shows a tooltip with the
cohort label, era or author metadata, and net line or commit count.
Clicking a band extracts it from the main graph into a separate line panel
below with its own $y$-axis; this is particularly useful for removing a
dominant author whose band would otherwise compress all others into an
unreadable sliver.
A double-click on the line panel restores all extracted cohorts. \par

\subsection{AI Use}
To speed up implementation, we used a few AI tools, primarily Claude
Haiku/Sonnet/Opus 4.x, as well as Qwen 3.6 running locally for smaller changes and
refactors. We validated these results with hands-on testing, in addition to AI-written
validation scripts. These were mostly used to address persistent regression bugs, e.g.,
during development of the commit graph we created a few test scripts that would
allow Claude to verify that commits on different branches were being correctly
separated after every change, as this would frequently break. \par
We also read
the code and manually directed Claude to address generic antipatterns we noticed, like
redundant styling and duplicated logic, as well as logical mistakes more specific
to our application, like using the wrong git flags or attempting to infer data when it was
explicitly available. No AI was used on the report or proposals for this
project, with the exception of one pass with the prompt `make light copy edits
to this document'\footnote{The unedited version of this report is included in the project
files}.

\subsection{Technical Challenges}
We initially tried to get all
the information we needed from the git log command; however, this involved manually parsing
commit messages to try and infer things like merges, which was not completely
accurate. In the end, we settled on the approach of getting the commit hashes
from the log and running a git command for every commit to get the additional
data we needed. This
approach was much slower but more reliable, and since the processing script only
needs to run once, the slowdown was acceptable. \par
A feature we attempted to implement but ultimately failed at was some kind
of `condensed commit' view that would allow us to support larger repositories.
The idea was that large numbers of commits tended to lag the app, so in
order to support larger repositories, we would want to combine 20 or so individual
commits into one. This proved to be very tricky, as doing so would cause us to
lose merges and branches and created weird edge cases in the commit graph. We
attempted strategies to mitigate this, like only combining commits on the
same branch or `smartly' combining the parents and children of the
commits we merged, but were ultimately unable to make this work. \par

\subsection{Selected Dataset(s)}
We selected 4 medium-size repositories to examine in this project. As addressed above,
we were unable to support very large repositories, but the four we selected gave us a
diverse sampling of git practices.
\begin{itemize}
    \item Flask (Python web server library) - A repository with one primary
    contributor with both long-running and short-lived feature branches.
    \item Click (Python CLI library) - A companion library to Flask with similar
    development patterns and a concentrated contributor base.
    \item HTTPX (Python HTTP client) - A modern async-capable HTTP library showing
    more distributed authorship and frequent dependency updates.
    \item Requests (Python HTTP library) - A mature, widely-adopted library with
    a long history and diverse contributor patterns over multiple maintenance phases.
\end{itemize}
\subsection{Expected results and evaluation methodology}

\noindent\textbf{Expected results.}
The primary deliverable of this project is a deployed web application
demonstrating the four visualizations described above against four real-world
Python repositories (Flask, Click, Requests, HTTPX).
Each visualization targets one or more of the research questions posed in
Section 1; we state our concrete expectations below.

\begin{itemize}
    \item \textit{Stabilized Sediment Graph — author mode (RQ1).}
        The commit graph's author filter answers RQ1 at the level of individual
        commits, but the sediment graph in author mode answers it at the level of
        \emph{surviving impact}: each band's width encodes how many lines written
        by a given author are still present in the codebase today.
        This distinguishes authors who made large early contributions that were
        later refactored away from those whose code forms the lasting foundation
        of the project—a distinction raw commit counts cannot make.
        The extract-sediment interaction reinforces this: pulling out a dominant
        author's band reveals the relative contributions of everyone else,
        turning a graph that might otherwise be unreadably skewed into one that
        communicates the full contributor hierarchy.
        We expect a small number of ``core'' contributors to dominate each
        repository, consistent with the power-law distributions reported in the
        Git-Truck literature.

        \begin{figure}[H]
            \centering
            \includegraphics[width=0.8\textwidth]{sediment-main-contributor.png}
            \caption{Implementation of the Stabilized Sediment Graph showing the outlier David Lord. Using this information, users can recognize that David Lord is the main contributor of the Flask repository.}
            \label{fig:sediment-main-contributor}
        \end{figure}

    \item \textit{Stabilized Sediment Graph — age-of-code mode (RQ3).}
        The age-of-code view should show that most surviving lines in mature
        repositories were written in early cohorts, with recent cohorts
        contributing smaller but growing fractions—a signature of stable,
        incrementally maintained projects.
        We additionally expect the sediment graph and word cloud to jointly
        surface a visible inflection point—an era in which commit subjects begin
        containing AI-assistance markers such as ``co-authored-by'' or tool
        names—providing evidence for or against RQ3.

        \begin{figure}[H]
            \centering
            \includegraphics[width=0.6\textwidth]{sediment-ai-usage.png}
            \caption{Implementation of the Stabilized Sediment Graph showing outlier lines of code additions in certain time frames. Users can recognize that there is likely AI usage beginning in 2024.}
            \label{fig:sediment-ai-usage}
        \end{figure}

    \item \textit{Word Cloud with tf-idf weighting (RQ2).}
        Restricting the time window to a known development phase (e.g.\ a major
        version release) should produce a cloud whose dominant terms closely
        match the human-readable changelog for that release.
        The tf-idf weighting should suppress generic filler words and surface
        release-specific terminology or contributor handles that plain frequency
        counts would bury.

    \item \textit{File Cluster Map (RQ4).}
        Tightly coupled subsystems (e.g.\ test files co-modified with their
        implementation counterparts) should form visually distinct clusters in
        the force-directed layout.
        Animating the graph over time should show the emergence of new modules
        and the decoupling or deprecation of old ones, giving developers an
        intuitive picture of architectural evolution.

        \end{figure}
\end{itemize}

\noindent\textbf{Evaluation methodology.}
We evaluate our visualizations through an informal expert survey.
Participants are professional software engineers recruited from the authors'
networks; each was shared a link to the hosted application and asked four
open-ended questions:

\begin{enumerate}
    \item What git tools do you currently use to understand development history
        (e.g.\ IDE integrations, \texttt{git log})?
    \item What questions are most important to you before beginning work on an
        unfamiliar project?
    \item Do you think a visual analytic system would be helpful for getting an
        overview of a git repository?
    \item Do any of the visualizations from the project help you answer the
        questions from (2)?  Why or why not?
\end{enumerate}

Questions 1 and 2 establish a baseline of the tools and information needs
participants already have, providing context for interpreting their reactions
to our system.  Question 3 gauges overall receptiveness to visual analytics
for git data, and question 4 directly probes whether our specific
visualizations address real developer needs.  Responses are analyzed
qualitatively: we look for recurring themes across participants and note
which visualizations are cited as useful or insufficient relative to the
needs surfaced in question 2.

\subsection{Evaluation results.}
We collected responses from four professional software engineers.
Participant A primarily uses Azure DevOps / TFS rather than raw git tooling,
and their most-used feature is a file-history view showing the change history
of individual files.
Participant B is a command-line-oriented git user whose go-to command is
\texttt{git log -{}-oneline -{}-graph -{}-all -{}-decorate} and does not use
IDE integrations or other visual tools. Participant C said they exclusively
used git tools from the IntelliJ IDE. Participant D did this as well as used more basic git
logging.

On question 2, the first three participants converged on the same underlying concern
despite different framings: they want to understand \emph{what code is
coupled} and \emph{how it has changed} before touching any of it.
Participant A asked specifically when and how the area they would be working
on changed, and which other areas might be affected by their changes.
Participant B framed this in terms of inferring functional and non-functional
requirements, including thinking through the implications for future
developers and maintainers. Participant D focused more on understanding the
culture of the repository, saying they prefer to contact key maintainers before
working on anything other than very small changes.

All four participants were receptive to visual analytics (question 3).
Participant A noted that it provides a quick understanding of what is
happening in a repo rather than ``staring at a wall of text.''
Participant B said it would be ``amazing.''

On question 4, the \textit{File Cluster Map} was cited by the first two participants
as the most directly useful visualization relative to their stated needs.
Participant A called it the closest to answering question 2 and said it
``would give a visual understanding of what areas of the code to pay
attention to when starting to work.''
Participant B highlighted the \textit{Commit Graph} as feeling like the
most important visualization overall, while also noting that the File Cluster
Map was valuable; they observed that the commit graph does not directly answer
question 2 but is nonetheless highly useful in practice.
However, participants C and D favored the \textit{Sediment Graph} as
the most useful overall visualization. They said that the authorship
filters would be most helpful for identifying key contributors.
Participant C liked how the sediment graph could be used to identify the current
active contributors as opposed to historical contributors who may have
contributed heavily early in the project's life cycle but were no longer active.
Participant D recommended that it would be helpful to see the sediment graph of a particular
set of files to help identify key contributors within certain modules.


The four responses are consistent with each other and with our expectations:
the File Cluster Map and Sediment Graph address the most practically urgent developer
questions: coupling and impact analysis for the File Cluster Map and
identifying key contributors for the Sediment Graph. Which of the two graphs
they favored broke down along the lines of which of the two question areas they
were most concerned about. In addition, multiple respondents mentioned the Commit
Graph as a useful structural overview.

\subsection{Discussion.}

The visualizations found in this project attempt to capture both syntactic and
semantic meaning from the repositories. The File Cluster Map and Sediment Graph
both attempt to capture information surrounding the code. They include file
connections, authorship, lines of code, edit dates, etc. The Word Cloud
and Commit Graph both provide semantic meaning from commits. These capture
information about the code.

Capturing semantic meaning is extremely difficult, especially visually.
Aggregating semantic data and producing a visualization is possible, but it requires
a representation of meaning that is both compact and comparable across commits.
Commit messages are a natural source of semantic signal, but their quality varies
enormously: terse or auto-generated messages (e.g.\ ``fix'' or ``update'') carry
little information, while detailed messages can surface genuine intent.
Our tf-idf-weighted Word Cloud partially addresses this by suppressing
high-frequency filler terms and surfacing distinctive vocabulary, but it cannot
recover meaning from commit messages that were never written expressively in
the first place.

Even works that perform deeper per-commit analysis remain grounded in
syntactic elements.
\cite{changeprism}, for instance, visualizes code changes through structural
diffs and code-level comparisons.
True semantic understanding of \emph{why} a
change was made or what conceptual role a module plays would require
language-model embeddings, which are beyond the scope of what git metadata
alone can provide.

\noindent\textbf{Limitations.}
Several limitations constrain the current system.
First, the preprocessing pipeline caps ingestion at 4500 commits for
performance reasons; very large repositories will be analyzed only partially.

Second, our evaluation sample is small (four participants), so the qualitative
findings should be treated as exploratory rather than conclusive.

Third, the project is not apt at capturing many syntactic elements that are 
found in Git-Truck. For example, the nested structure of files and directories
is not sufficiently captured in the File Cluster Map.

All visualizations operate exclusively on git metadata; no static
analysis or code parsing is performed, so structural insights are inferred
from co-change patterns rather than from the code itself.

\noindent\textbf{Future work.}
Integrating language-model embeddings of commit messages could replace the
tf-idf heuristic with a richer semantic clustering that groups commits by
topic even when individual words differ. One example is Andrej Karpathy's
LLM Wiki \href{https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f}{link}.

A larger, more systematic user study with participants from diverse
development backgrounds (embedded systems, data science, large enterprise)
would strengthen the evaluation conclusions.

Finally, making the preprocessing pipeline incremental (updating JSON files
as new commits arrive rather than rebuilding from scratch) would enable the
tool to be used on live repositories without the current one-time processing
overhead.
